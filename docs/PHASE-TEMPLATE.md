# Phase 詳細ファイル テンプレート

Phase C（実装準備）で Opus が `docs/phases/phase-N-xxx.md` を作成する際のテンプレート。
Sonnet が「このファイルだけ読めば実装できる」レベルの具体性を目指す。

---

## テンプレート

```markdown
# Phase N: タイトル（X.X人日）

**ゴール：** このPhaseが完了した時に何が動くか、1行で。

**前提：** 着手に必要な条件（前Phase完了など）。

**参照仕様:**
- `docs/xxx-spec.md` §N — 関連セクション

---

## タスク一覧

| # | タスク | 成果物 |
|---|--------|--------|
| N-1 | タスク名 | `path/to/file` |
| N-2 | タスク名 | `path/to/file` |

---

## N-1: タスク名

**ファイル:** `path/to/file`

### 実装内容

- 何を実装するか、箇条書きで
- 関数シグネチャ・引数・戻り値を明記
- 既存コードとの接続点（どのクラスから呼ばれるか等）

### コード（参考実装 or 骨格）

Sonnet が「このまま書ける」レベルのコードを提示する。
全行書く必要はないが、構造・命名・パターンが分かる程度には書く。

\```php
// 例: クラス骨格
class Example {
    public function method( string $param ): array {
        // TODO: 実装内容の説明
    }
}
\```

### テストケース（該当する場合）

| テスト | 内容 |
|--------|------|
| `test_xxx` | 何を検証するか |

---

## 完了条件

- [ ] 検証可能で具体的な条件
- [ ] 「動く」ではなく「何が・どう動く」で書く

## コミット粒度の目安

1. 成果物グループ → 「コミットメッセージ例」
2. 成果物グループ → 「コミットメッセージ例」
```

---

## 書き方ガイドライン

### 必須セクション

| セクション | 目的 |
|-----------|------|
| ゴール | Sonnet が完了判断できる |
| 前提 | 依存関係の明示 |
| タスク一覧 | 全体像の把握 |
| 各タスク詳細 | 迷わず実装できる |
| 完了条件 | Phase完了のチェックリスト |
| コミット粒度 | コミット単位の統一 |

### コードの書き方

- **骨格レベル以上**: クラス定義・メソッドシグネチャ・主要ロジックのフローを書く
- **設定ファイル**: phpunit.xml, phpstan.neon 等はそのまま使える完成形を書く
- **既存パターン踏襲**: プロジェクト内の既存コードと命名・構造を揃える例を示す
- **TODO コメント**: Sonnet に判断を委ねる部分は `// TODO:` で意図を明記

### やってはいけないこと

- ゴールが曖昧（「いい感じに」「適切に」）
- 成果物パスが未定（「どこかに作る」）
- コードなしでテキスト説明のみ（Sonnet の解釈ブレが大きくなる）
- 1タスクに複数ファイルを詰め込みすぎる（コミット粒度が崩れる）

### PHASE-PLAN.md との関係

```
docs/PHASE-PLAN.md          ← 目次 + 全体設計（クラス図・ディレクトリ構成）
docs/phases/phase-N-xxx.md  ← 各Phaseの実装詳細（このテンプレートで作成）
```

- PHASE-PLAN.md の「実装フェーズ一覧」テーブルからリンクする
- Phase詳細には全体設計を重複して書かない（PHASE-PLAN.md を参照）
