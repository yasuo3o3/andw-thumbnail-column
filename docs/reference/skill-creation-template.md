# スキル作成テンプレート（全流派良いとこ取り）

> 新しいスキルを作るときにこのテンプレートをコピーして使う。
> 分析の根拠: [skill-patterns-analysis.md](discussion/skill-patterns-analysis.md)
>
> **運用方針**: 初回はOpusで作成しながらテンプレート自体を改善する。
> 2回目以降はSonnetでも同品質のスキルが作れることを目指す。
> Opusが判断を要した箇所は、テンプレートに判断基準として追記すること。

---

## このテンプレートの設計方針

| 採用元 | 採用した要素 | 理由 |
|--------|-------------|------|
| 流派A（WP公式） | Escalation セクション | 「困ったらこうする」が実用的 |
| 流派B（wshobson） | 500-700行、コード例豊富、❌→✅対比 | インストール数最多の実績 |
| 流派C（jeffallan） | Role Definition、MUST DO/DON'T | ガードレールとして効果的 |
| 流派D（bobmatnyc） | progressive_disclosure（Quick Start） | 最初の一歩が分かりやすい |

---

## テンプレート本体

以下を `SKILL.md` としてコピーし、`{...}` 部分を埋める。

````markdown
---
name: {skill-name}
description: |
  {1行の概要}。
  Use when {トリガー条件1}, {トリガー条件2}, or {トリガー条件3}.
metadata:
  author: {GitHub URL}
  version: "1.0.0"
  domain: {language | framework | toolchain | workflow}
  triggers:
    - {キーワード1}
    - {キーワード2}
    - {キーワード3}
  role: {specialist | guide | validator}
  scope: {implementation | review | documentation}
  related-skills:
    - {関連スキル1}
    - {関連スキル2}
---
````

## 使い方ガイド（Sonnet でも実行可能な手順）

### ステップ0: 情報収集（調査フェーズ）
> **Opus判断メモ**: 初回作成時、ここが最も判断を要した。Sonnet向けに手順化する。

1. **公式ドキュメントを WebSearch/WebFetch で収集する**
   - 公式サイト（docs.x.com 等）は 403 でブロックされることがある → 記事サイト経由で情報を集める
   - 最低3つの独立したソースから同じ数値を確認する（1つのソースだけを信じない）
2. **既存スキルの構造を確認する**
   - `~/.claude/skills/` に関連スキルがないか確認
   - 既存スキルとの棲み分けを明確にする（重複を避ける）
3. **収集した情報の信頼度を評価する**
   - 公式発表 > テック系メディア > 個人ブログ
   - 料金などの具体的数値は最低2ソースで一致を確認
   - 「Developer Console でしか確認できない」情報は、その旨を明記する

### ステップ1: スコープを決める
- 「同時に必要な知識」の塊を1スキルにする
- 大きすぎたら分割、小さすぎたら統合

### ステップ2: description を最初に書く
- `Use when` + 実際にユーザーが言いそうな言葉をトリガーにする
- これが Claude Code の読み込み判断基準になる

### ステップ3: Core Patterns を書く
- ❌→✅ の対比を基本形とする
- 各パターンは「名前 + いつ使う + コード + 理由」の4点セット
- 目安: 5-10パターンで SKILL.md 全体が 500-700行
- **コード例の言語選択**: ターゲットユーザーが最も使う言語を選ぶ。迷ったら JavaScript（Web開発者に最も届く）
- **パターンの粒度**: 1パターン = 1つの判断ポイント。「プラン選択」「認証方式選択」「バッチ取得」のように、開発者が実際に迷う場面ごとに切る

### ステップ4: ガードレールを書く
- MUST DO / MUST NOT DO は実体験から抽出する
- 「公式ドキュメントに書いてないが実際に踏んだ罠」が最も価値が高い

### ステップ5: ドメイン固有セクションを追加する
> **Opus判断メモ**: テンプレートの固定セクション以外に、ドメイン固有の情報セクションが必要になることが多い。

- 料金・課金系 → 「料金体系の全体像」テーブル
- API系 → 「セットアップ手順」ステップバイステップ
- フレームワーク系 → 「バージョン互換表」
- これらはテンプレートの Core Patterns の前に配置する

### ステップ6: 実測検証でスキルの主張を裏取りする
> **v1.2追加**: X API スキルで「公式料金 vs 実測コスト」に大きな乖離を発見。検証なしでは誤った情報をスキルに残すリスクがある。

1. **スキルの主要な数値的主張をリストアップ**
   - 料金、パフォーマンス、制限値など、ユーザーの判断に影響する数値
2. **低コストのテスト計画を設計する**
   - 予算を決める（例: $1〜3）
   - 1つの変数だけを変えて比較するテスト（A/B的）
   - 手動確認ポイントを入れる（Developer Console の残高等）
3. **テストを実行し、公式 vs 実測の対比表を作る**
   - 一致 → スキルに「実測確認済み」と明記
   - 乖離 → スキルを修正し、発見した実際の挙動を記載
4. **テスト中に発見した「公式に書いていない挙動」をスキルに追記**
   - 例: 重複排除の挙動、レート制限の体感、課金タイミング
5. **バージョンを上げる**（v1.0 = 公式ベース → v1.1 = 実測反映）

**実測検証セクションのテンプレート**:
```markdown
### 実測検証結果（YYYY-MM-DD、$X.XX消費）

| 操作 | 公式 | 実測 | 備考 |
|------|------|------|------|
| {操作1} | {公式の値} | **{実測値}** | {一致/乖離の説明} |
```

### ステップ7: references/ が必要か判断する
- 500行以内に収まる → 単体ファイルで完結（流派B方式）
- 700行を超える → references/ に分割（流派A/C方式）
- references/ を使う場合は SKILL.md に Reference Guide テーブルを置く

### 行数の目安

| レベル | SKILL.md | references/ | 合計 | 向いているケース |
|--------|----------|-------------|------|-----------------|
| 小 | 300-450行 | なし | 300-450行 | 狭いドメイン（1つのAPI等）。実例: x-api-pay-per-use (v1.0: 376行 → v1.1: 417行、実測検証で増加) |
| 中（推奨） | 500-700行 | なし | 500-700行 | 標準的な技術領域 |
| 大 | 100-200行 | 3-5ファイル | 1,000-3,000行 | 広いドメインを分割 |

## チェックリスト（公開前）

- [ ] description に `Use when` + トリガーワード3つ以上
- [ ] Quick Start で3行以内に「何ができるか」が分かる
- [ ] コード例が5個以上ある
- [ ] ❌→✅ 対比が3組以上ある
- [ ] MUST DO / MUST NOT DO がそれぞれ3個以上
- [ ] 行数が 300-700行の範囲内（小: 300-400、中: 500-700、大: references分割）
- [ ] related-skills で関連スキルを参照している
- [ ] 情報ソースを最低2つで照合済み（特に料金・数値データ）
- [ ] ドメイン固有セクション（料金表/セットアップ手順等）が必要なら追加済み
- [ ] 数値的主張に実測検証があるか（料金・制限値など。未検証なら「未検証」と明記）
- [ ] テスト中に発見した「公式に書いていない挙動」を反映済み

## Anthropic skill-creator から学んだベストプラクティス

> 出典: [anthropics/skills/skill-creator](https://github.com/anthropics/skills/tree/main/skills/skill-creator)
> ツール: `_claude-config/skills/skill-creator/` にインストール済み

### description の書き方（最重要）

descriptionはスキルのトリガー判断に使われる唯一のメタデータ。

1. **「少し積極的（pushy）に」書く** — Claudeはスキルを under-trigger しがちなので、「このキーワードが出たら必ず使え」くらいの強さで書く
2. **Use when を明示する** — ユーザーが実際に言いそうなフレーズを列挙する
3. **近接スキルとの区別を明記する** — 「Xとは異なり、こちらはYスタイル」
4. **悪い例**: `"Format this data"` / **良い例**: `"ok so my boss sent me this xlsx file..."`（具体的で長い）

### スキル本体の書き方

1. **ALWAYS/NEVER の多用はNG** — 代わりに「なぜそうすべきか」を説明する。LLMは理由を理解すれば正しく振る舞う
2. **500行を超えそうなら references/ に分割** — SKILL.md本体は常にコンテキストに載るので短く保つ
3. **テストケースからの一般化を意識する** — 特定の例にoverfitせず、汎用的に書く

### テスト→改善ループ（skill-creatorの中核機能）

```
ドラフト作成 → テストケース2-3件作成 → 実行 → eval-viewerでレビュー → 改善 → 繰り返し
```

- テストケースは「本物のユーザーが実際に言いそうなこと」を書く
- ベースライン（スキルなし）と比較して効果を測定する
- `_claude-config/skills/skill-creator/eval-viewer/generate_review.py` でHTMLレビュー画面を生成

### description最適化（自動チューニング）

```bash
python -m scripts.run_loop \
  --eval-set trigger-eval.json \
  --skill-path path/to/skill \
  --model claude-opus-4-6 \
  --max-iterations 5
```

- should-trigger / should-not-trigger のテストクエリを20件作成
- 60% train / 40% test に分割して過学習を防ぐ
- 各クエリ3回実行して信頼性のあるトリガー率を算出

## フィードバック履歴

### v1.1（X API スキル作成時・Opus）

1. **テンプレートの固定構造だけでは足りない**: 料金体系やセットアップ手順など、ドメイン固有のセクションが必要 → Domain Context セクションを追加
2. **情報収集フェーズが最も判断を要する**: 公式サイトが403で取得できない、ソースごとに数字が微妙に異なる等 → ステップ0として手順化
3. **376行で十分機能した**: 狭いドメインでは500行に無理に伸ばす必要なし → 行数目安を「小: 300-400行」に調整
4. **コード例の言語選択は迷う**: JavaScript を選んだが、Python やcURL が適切な場合もある → 判断基準を明文化

### v1.3（skill-creator導入・2026-03-05）

1. **Anthropic公式skill-creatorをインストール**: `_claude-config/skills/skill-creator/` に配置。スキルの新規作成・改善・description最適化をサポート
2. **descriptionの書き方をベストプラクティスに基づき改訂**: 「pushy に書く」「Use when を明示」「近接スキルとの区別」の3原則を追加
3. **既存スキル（yasuo-blog-writer, ai-counselor）のdescriptionを改善**: トリガー条件の明示と相互の棲み分けを追記
4. **テスト→改善ループのセクションを追加**: eval-viewer、ベースライン比較、description自動最適化の手順を文書化

### v1.2（X API スキル実測検証後・2026-03-01）

1. **公式ドキュメントの数値と実際の挙動が大きく異なることがある**: Search は「$0.005/resource」と書いてあるが、実測では「$0.005 × 返却件数」だった。公式だけを信じたスキルは誤情報を含むリスクがある → ステップ6「実測検証」を追加
2. **テストで踏んだ地雷が最も価値の高いスキル内容になる**: レート制限を体験→スキルの Escalation に追記、重複排除を確認→Pattern を「実測確認済み」に更新。テストなしでは書けなかった実践的な知識
3. **バージョニング戦略が明確になった**: v1.0 = 公式ドキュメントベース、v1.1 = 実測検証を反映。この2段階で「最初に出す」と「精度を上げる」を分離できる
4. **チェックリストに「実測検証」を追加**: 数値的主張がある場合は検証済みか、未検証なら明記すべき
