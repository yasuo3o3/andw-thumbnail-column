# Design Review (Codex -- 5 Personas)

## 1. セキュリティ専門家

### 指摘事項
- HIGH: `render_thumbnail_column()` で `get_the_post_thumbnail()` を同一リクエスト内で2回呼ぶ設計（説明文上の実装例）は、将来フィルターフック連携時に意図しないHTML混入面を広げる。1回だけ取得して検証・出力する形に固定した方が安全。
- MEDIUM: `manage_options` 固定は設定変更権限としては安全側だが、運用上は編集者ロールが対象ユーザー定義に含まれているため、管理者へ権限集中が起きる。要件と実装方針の不一致は「権限の迂回運用（共有アカウント化）」を招きやすく、結果としてセキュリティリスクになる。
- MEDIUM: サニタイズは保存時に定義されているが、読み出し時に再検証してからフック登録する記述が弱い。DB改ざんや不正移行データを想定し、`admin_init` でも whitelist 再評価を必須化した方がよい。
- LOW: nonce対策は Settings API 前提で妥当。ただし将来AJAX追加時の方針が未記載。拡張余地として「非同期処理追加時は action別 nonce を必須化」を明文化すると保守時の事故を防げる。

### 良い点
- `settings_fields()` によるCSRF対策、`current_user_can()` 再チェック、whitelistサニタイズ、`ABSPATH/WP_UNINSTALL_PLUGIN` ガードが明文化されている。
- 出力時エスケープ方針（`esc_attr` / `esc_html` / `wp_kses_post`）が具体的で、OWASP Top 10 の XSS/Injection 対応に直結している。

## 2. WordPress熟練者

### 指摘事項
- HIGH: 固定ページは一覧フック名が `manage_pages_columns` / `manage_pages_custom_column` 系で、`manage_{$post_type}_posts_columns` だけでは `page` が漏れる可能性が高い。要件「post/page対応」と矛盾するためフック戦略を再設計すべき。
- MEDIUM: `admin_init` で毎回動的に複数フックを登録するのは機能上成立するが、WP管理画面全体で走る。`current_screen` 連携または対象画面限定で登録すると他プラグイン共存時に副作用が減る。
- MEDIUM: 列を「末尾固定(priority 99)」は意図として明確だが、他プラグインも同じ戦略を取るため順序競合が起きる。フィルタで挿入位置を選べる拡張ポイントを設けると互換性が高い。
- LOW: テキストドメインは記載されているが、ラベル文字列のi18n関数（`__()`, `esc_html__()` など）の方針が未記載。配布前提なら必須。

### 良い点
- Settings API を正しく使う前提で設計され、`submit_button()` などWP標準UIに寄せている。
- オプション名・クラス名・定数プレフィックスの統一方針があり、名前衝突を避けやすい。
- `wp_add_inline_style()` を使う方針はWP流儀に沿っている。

## 3. 初心者ユーザー

### 指摘事項
- HIGH: 対象ユーザーに「編集者」を含むのに、設定画面が `manage_options` 前提だと編集者は自分でオン/オフできない。期待と実際の操作可能範囲がズレて混乱する。
- MEDIUM: 設定画面で「ラベル名のみ表示」は分かりやすい一方、同名ラベルのカスタム投稿タイプがあると識別できない。補助情報（例: 小さくスラッグ表示）の検討が必要。
- MEDIUM: 全チェックOFFで全非表示は仕様として妥当だが、初心者には「壊れた」と見えやすい。説明文や注意メッセージが必要。
- LOW: プレースホルダーが単なるグレー四角だと意味が伝わりにくい。`aria-hidden` はあるが、視覚的に「画像なし」と分かるUI補助があると親切。

### 良い点
- 設定導線が「設定 → サムネイル列」で直感的。
- 投稿タイプ自動検出で、初心者が技術用語を知らなくても選択できる。

## 4. 壊し屋

### 指摘事項
- HIGH: 「画像ファイル消失時もplaceholder」と書かれているが、実装案は `_thumbnail_id` の有無中心で、添付メタ破損・中間サイズ欠損・外部ストレージ遅延などの異常系判定が曖昧。`get_the_post_thumbnail()` が空文字を返すケースを主判定にすべき。
- HIGH: 大量CPT（数十以上）環境で `admin_init` たびにループ登録すると管理画面の負荷増大が起きうる。負荷試験条件（投稿タイプ数・投稿件数）を確認項目へ追加すべき。
- MEDIUM: `add_option()` で初期値を入れるだけだと、マルチサイトやオプション破損後の自己修復が弱い。読み出し時のデフォルトフォールバックを厳密化した方が壊れにくい。
- MEDIUM: CSSが固定幅60pxのみで、高DPI・狭い管理画面・他列過多時にレイアウト崩れが起きる可能性。最小幅や折返し制御を検討した方がよい。

### 良い点
- 「全チェックOFF」「画像ファイル消失」「再インストール復元」など異常寄りの確認項目が既に入っている。
- uninstallでオプション削除するため、再導入時の不整合が残りにくい。

## 5. AIクローラー

### 指摘事項
- MEDIUM: 本機能は管理画面限定でフロント非影響だが、その設計意図（SEO非影響）を明文化しておくと誤解を避けられる。
- LOW: 構造化データ・メタタグ・OGPには直接作用しないため、SEO改善効果を期待する説明があるとミスマッチになる。README/仕様で「編集効率改善プラグイン」と明確化すべき。
- LOW: 管理画面のサムネイル表示が増えることでクロール負荷は基本無関係だが、メディアURL取得失敗時の管理画面パフォーマンス低下は編集体験を悪化させ、間接的に更新頻度へ影響し得る。監視指標（管理画面表示速度）を補助的に持つとよい。

### 良い点
- `is_admin()` ガードにより公開ページHTMLを変えないため、検索順位やインデックス挙動への副作用がほぼない。
- 構造化データを壊す要素が設計上含まれていない。

## 総評
- セキュリティと基本設計は良好だが、最優先は「`page` 用フック差異への対応」と「対象ユーザー（編集者）に対する権限設計の再整合」。
- 次点で、異常系判定（画像欠損時）と高負荷時の登録戦略を詰めると、実運用で壊れにくい設計になる。
