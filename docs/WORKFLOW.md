# 開発ワークフロー — Opus/Sonnet/Codex 分業ルール

## 概要

Opus が設計・レビューを担当し、Codex が実装を担当する分業体制。
**Opus は原則としてコードを書かない。** 設計書・レビュー結果・プロンプトを書く。

### モデル割り当ての原則

**前工程と異なるモデル** が次工程を担当することで、視点の切り替えを保証する。

| フェーズ | 担当モデル | 理由 |
|---------|-----------|------|
| A. 設計相談 | **Opus** | 設計判断 |
| B. ペルソナレビュー | **Sonnet** | Opus の設計を別の目で見る |
| C. 実装準備 | **Opus** | 設計→実装の橋渡し |
| D. 実装 | **Codex（gpt-5.3-codex）** | 自律的な反復実装に特化、コスト効率が高い |
| E. コードレビュー | **Sonnet（5並列）+ Opus（統合）** | Sonnet がペルソナレビュー、Opus が統合・判定 |
| E'. 修正サイクル | **Codex → Opus** | Codex が修正、Opus が単独確認 |
| F. テスト・ユーザー検証 | **Opus + オーナー** | テストIssue作成→実機確認 |

---

## 要求定義の伝達ルール

すべての判断の起点は **要求定義** である。仕様書（`docs/SPEC.md`）に以下の5項目が定義される:

| # | 項目 | 問い |
|---|------|------|
| 1 | 対象ユーザー | 誰の困りごとを解決するのか |
| 2 | 起点の課題 | どんな未解決の問題があるのか |
| 3 | 理想の行動変化 | ユーザーにどうなってほしいのか |
| 4 | 事業上の狙い | なぜ今これを解決するのか |
| 5 | 成功指標 | 何をもって成功とするのか |

**下流への伝達:**
- Phase A: PHASE-PLAN.md の冒頭に要求定義を転記する
- Phase C: 各Phase詳細ファイル（`docs/phases/phase-N-xxx.md`）の冒頭に、そのフェーズが満たすべき要求定義項目を抽出して含める
- Phase D: Codex は実装判断に迷ったとき、要求定義に立ち返って判断する
- Phase E: レビュー時に「要求定義を満たしているか」を評価観点に含める

要求定義が不明確な場合は、実装に入らず Phase A に差し戻す。

## フェーズ定義

```
A. 設計相談  →  B. ペルソナレビュー  →  C. 実装準備  →  D. 実装  →  E. レビュー  →  E'. 修正  →  F. テスト
   (Opus)          (Sonnet並列)           (Opus)        (Codex)    (Sonnet+Opus)  (Codex→Opus)   (Opus+オーナー)
```

### フェーズ A — 設計相談（Opus + オーナー）

**前提:** `docs/SPEC.md`（仕様書）が存在すること。なければ `/spec-prep` で先に作成する。

**やること:**
- `docs/SPEC.md` の要求定義・要件定義を読み込む
- オーナーと技術設計を議論
- `docs/PHASE-PLAN.md` に計画草案を作成（冒頭に要求定義を転記）
- 不明点があれば AskUserQuestion で確認

**出力:** PHASE-PLAN.md 草案（要求定義付き）

**📋 成果物の提案（任意）:**
- 「概算見積もりを作成しますか？」→ `/estimate`（企画段階モード）
- 「契約書のたたき台を作成しますか？」→ `/contract-draft`

**⛔ GATE-A: オーナーが「OK」「承認」等と言っても、コーディングを開始してはならない。**
- 次のアクション: 「ペルソナレビュー（Phase B）に進みます」と宣言し、Phase B に移行
- HANDOFF.md の `workflow_phase` を `B` に更新

---

### フェーズ B — ペルソナレビュー（Sonnet 並列エージェント）【任意】

**やること:**
- 5人のペルソナ（security-expert, wp-expert, beginner-user, breaker, ai-crawler）を **Sonnet エージェント** で並列起動し、設計をレビュー
- `/review-design` コマンドで実行可能
- Opus がレビュー結果を統合し、PHASE-PLAN.md を改訂

**出力:** 改訂版 PHASE-PLAN.md

**⛔ GATE-B: オーナーが改訂版を承認しても、コーディングを開始してはならない。**
- 次のアクション: 「Sonnet 向けに実装準備（Phase C）を行います」と宣言し、Phase C に移行
- HANDOFF.md の `workflow_phase` を `C` に更新

---

### フェーズ C — 実装準備（Opus）

**やること:**
1. `docs/PHASE-TEMPLATE.md` に従い、Phase 詳細ファイルを作成
   - `docs/phases/phase-N-xxx.md` に各Phaseの実装詳細を記述
   - **各ファイルの冒頭に、そのフェーズが満たすべき要求定義を SPEC.md から抽出して含める**
   - 各タスクに具体的なファイルパス・関数シグネチャ・コード骨格を明記
   - 既存コードのパターン（命名規則、IIFE構成等）を明示
   - `docs/PHASE-PLAN.md` の一覧テーブルからリンクする
2. Sonnet 用スタータープロンプトを作成
   - `docs/SONNET-PROMPT.md` に保存
   - 含む内容: 作業範囲、参照ファイル、コーディング規約、コミット粒度
3. Notionガントチャートに初期登録（詳細は「Notionガントチャート連携」セクション参照）

**出力:** Phase 詳細ファイル + SONNET-PROMPT.md + ガントチャート登録

**📋 成果物の提案（任意）:**
- 「詳細見積もりを作成しますか？」→ `/estimate`（計画段階モード）

**⛔ GATE-C: オーナーが承認しても、Opus 自身がコーディングを開始してはならない。**
- 次のアクション: 「Codex 用プロンプトが準備できました。/codex スキルで実装を開始してください」と伝える
- HANDOFF.md の `workflow_phase` を `D` に更新

---

### フェーズ D — 実装（Codex、/codex スキル経由）

**実行者:** Claude（Opus/Sonnet）が `/codex` スキル経由で Codex に実装タスクを投げる

**初期セットアップ（初回のみ）:**
- `composer install` を実行（WPCS・PHPStan をインストール）
- `phpcs.xml.dist` の `PLUGIN-TEXT-DOMAIN` をプロジェクトのテキストドメインに変更
- `phpcs.xml.dist` のプレフィックス（`andw_` / `Andw_` / `ANDW_`）をプロジェクト固有値に変更

**Codex のルール:**
- `docs/PHASE-PLAN.md` のステップに従い実装
- PHASE-PLAN.md に記載された単位ごとにコミット
- 各ステップ完了時に HANDOFF.md を更新
- **各フェーズ内で自動テスト（PHPUnit等）を回し、コード上は動く状態を維持する**
- フロント確認・管理画面確認はこの段階では行わない（Phase F で一括実施）

**Opus は何もしない**（Codex が実行中のため）

---

### フェーズ E — コードレビュー（Sonnet 5並列 + Opus 統合）

**実行者:** Opus（統合・判定）+ Sonnet エージェント（レビュー実行）

**ペルソナ分担（全て Sonnet で並列実行）:**

| ペルソナ | 観点 |
|---------|------|
| beginner-user | 初心者が迷わず使えるか |
| ai-crawler | SEO・構造化データ・クローラビリティ |
| security-expert | セキュリティ脆弱性・入力検証・権限チェック（`docs/reference/WP-SECURITY-CHECK.md` 参照） |
| wp-expert | WordPress規約準拠・フック設計・互換性 |
| breaker | 想定外の入力・エッジケース・壊れるパターン |

**やること:**
1. Sonnet ペルソナ 5人を並列起動
2. Opus が全結果を統合し、BUG/FIX/IMP リストを作成
4. 要求定義を満たしているかを評価
5. 指摘事項をオーナーと確認し、修正範囲を確定
6. 修正が必要な場合 → Sonnet 用の修正プロンプトを作成

**出力:** 統合レビュー結果 + 修正プロンプト（必要に応じて）+ リリース判定

---

### フェーズ E' — 修正サイクル（Codex 修正 → Opus 単独確認）

**実行者:** Codex（修正）→ Opus（確認）

**やること:**
1. Codex が修正プロンプトに従い修正を実施（Phase D と同じ手順、`/codex` スキル経由）
2. Opus が修正箇所を **単独で確認**（ペルソナレビューは不要）
3. 修正が不十分な場合 → **Opus が直接修正する**（GATE ルール特例）

**直接修正ルール:** 一度 Codex に修正を依頼して確認した結果、まだ BUG が残っている場合は、再度 Codex に差し戻さず Opus が直接コードを修正する。修正→確認→再修正の往復は無駄が多いため。

**判断基準:** BUG が 0 件、FIX の残りが軽微であれば Phase F に進む。

---

### フェーズ F — テスト・ユーザー検証（Opus + オーナー）

**タイミング:** 全フェーズの実装・レビュー・修正が完了した後。Phase E のリリース判定が「合格」になってから。

**やること:**
1. `/create-test-issues` でテストIssueを作成する。以下の2種類に分ける:
   - **管理画面テスト**（assignee: yasuo3o3）: 管理者権限が必要な操作の確認。必要最小限の項目に絞る
   - **フロントエンドテスト**: Claude in Chrome等のAIエージェントで自動確認できる範囲のテスト
2. オーナーが管理画面テストを実機で確認
3. フロントエンドテストをAIエージェントで実行
4. 不具合があれば修正（直接修正ルール適用）
5. オーナーが **ユーザーとして** フロントを触る（要求定義の検証）
   - 「気軽に使えるか」「迷うポイントはないか」「目的の行動を取りたくなるか」を感覚で判断
   - これは動作テストではなく要求定義の検証である

**📋 成果物の提案（任意）:**
- 「実績ベースの見積もりを作成しますか？」→ `/estimate`（完成後モード）
- 「テスト結果報告書を作成しますか？」→ `/test-report`
- 「検収チェックリストを作成しますか？」→ `/acceptance-checklist`
- 「操作マニュアルを作成しますか？」→ `/user-guide`
- 「要件定義書を作成しますか？」→ `/spec-doc`

**出力:** テスト完了 + リリース可否の最終判断

## GATE 違反の検知ルール

Opus が以下のツールを呼び出そうとした場合、GATE 違反の可能性がある:

| 現在のフェーズ | 禁止アクション |
|--------------|--------------|
| A, B, C | Write/Edit で `.php`, `.js`, `.css` ファイルを変更 |
| A, B, C | Bash で `php`, `npm`, `composer` 等のビルド/実行コマンド |
| D | Opus セッションでのコード変更（Sonnet が別ウィンドウで作業中） |

**例外:**
- ドキュメントファイル（`.md`）の編集は全フェーズで許可
- `PHASE-PLAN.md`, `HANDOFF.md`, `CHANGELOG.md`, `SONNET-PROMPT.md` 等
- レビュー中の軽微な修正（typo、コメント修正）は Phase E で許可

## HANDOFF.md でのフェーズ追跡

HANDOFF.md の「現在地」セクションに以下の形式でフェーズを記録:

```
## 現在地

**ワークフローフェーズ: C（実装準備）**

Phase 3 の設計完了。Sonnet 向けプラン整形中。
```

フェーズ遷移時は必ず HANDOFF.md を更新し、Notionガントチャートのステータスも更新すること。

## フェーズ遷移フロー図

```
オーナー: 「こういう機能がほしい」
  ↓
[Phase A] Opus: 設計 → PHASE-PLAN.md 草案
  ↓ オーナー承認
  ↓ ⛔ コーディング禁止！
[Phase B] Opus: ペルソナレビュー（Sonnet並列）→ PHASE-PLAN.md 改訂 【任意】
  ↓ オーナー承認
  ↓ ⛔ コーディング禁止！
[Phase C] Opus: Sonnet向け整形 → SONNET-PROMPT.md 作成
  ↓ オーナー承認
  ↓ ⛔ コーディング禁止！
[Phase D] Codex（/codex スキル）: 実装 → コミット
  ↓ 単位ごとに完了
[Phase E] Sonnet 5並列（beginner/crawler/security/wp/breaker）→ Opus 統合・判定
  ↓ オーナーと修正範囲確認 → 修正プロンプト作成
[Phase E'] Codex（/codex スキル）: 修正 → Opus 単独確認
  ↓ BUG 残あり → E' 繰り返し（2回目以降は Opus 直接修正）
  ↓ BUG 0 → Phase F へ
[Phase F] テストIssue作成 → 管理画面テスト（やすお）+ フロントテスト（AI）
  ↓ 不具合あり → 修正（直接修正ルール）
  ↓ やすおユーザーテスト（要求定義の検証）
  ↓ 完了 🎉
```

## Notionガントチャート連携

### データベース情報
- **DB名**: プラグイン開発ガントチャート
- **data_source_id**: `1e1570a3-7c0c-4db3-a741-d18136b8f7b8`
- **操作方法**: Notion MCP経由でページを追加・更新

### 登録タイミング: Phase C（実装準備）

Phase C でフェーズ詳細が確定したら、以下を登録する:

**プロジェクト行（1行）:**

| フィールド | 値 |
|---|---|
| プロジェクト名 | リポジトリ名（andw-xxx） |
| タスク名 | プラグイン名 |
| 開始日 | プロジェクト全体の開始日 |
| 終了日 | プロジェクト全体の予定終了日 |
| 担当 | やすお |
| 種別 | プロジェクト |
| 優先度 | 1〜5（プロジェクト間の優先順位） |
| ステータス | 進行中 |

**フェーズ行（Phase A〜F ごとに1行）:**

| フィールド | 値 |
|---|---|
| プロジェクト名 | リポジトリ名（andw-xxx） |
| タスク名 | Phase A: 設計相談 等 |
| 開始日 | フェーズ予定開始日 |
| 終了日 | フェーズ予定終了日 |
| 担当 | Opus / Sonnet / やすお（フェーズによる） |
| 種別 | フェーズ |
| 優先度 | ―（フェーズに優先度はない。順番に進む） |
| ステータス | 未着手 / 進行中 / 完了 |

### 更新タイミング: 各フェーズ完了時

フェーズ遷移のたびに（HANDOFF.md 更新と同時に）:
1. 完了したフェーズのステータスを「完了」に更新
2. 次のフェーズのステータスを「進行中」に更新
3. 実績と予定にズレがあれば終了日を修正

### ビューの使い方
- タイムラインビューで「プロジェクト」グループ化
- フィルタ: 担当=やすお で「自分の出番」だけ表示可能
